# Git Workflow Guide

Manual guide for day-to-day git operations on this project.

---

## Branch Strategy

| Branch | Purpose |
|--------|---------|
| `main` | Stable production-ready code |
| `feature/<name>` | New features |
| `fix/<name>` | Bug fixes |
| `hoang_*`, `<name>_*` | Personal dev branches |

**Never commit directly to `main`.** Always work on a branch and open a PR.

---

## Daily Workflow

### 1. Start from an up-to-date main

```bash
git checkout main
git pull origin main
git checkout -b feature/my-feature
```

### 2. Make changes and stage specific files

```bash
# Stage specific files — avoid git add . or git add -A
git add src/models/registry.py
git add deployment/fastapi/web.py

# Review what you're about to commit
git diff --staged
```

### 3. Commit

Use present-tense, imperative descriptions. Explain WHY, not WHAT.

```bash
git commit -m "fix: save relative artifact paths to model registry

Previously the training pipeline saved absolute Windows paths,
breaking registry portability on other machines and in Docker."
```

### 4. Push to origin

```bash
git push origin feature/my-feature

# First push on a new branch
git push -u origin feature/my-feature
```

---

## Opening a Pull Request

```bash
# Using GitHub CLI
gh pr create \
  --title "fix: save relative artifact paths in registry" \
  --body "## Summary
- Fixed \`src/models/registry.py\` to convert absolute paths to POSIX-relative
- Prevents model_registry.json from containing Windows-only absolute paths

## Test plan
- [ ] Run training pipeline, verify model_registry.json has relative paths
- [ ] Start API server, verify /model-info loads correctly"
```

Or use the GitHub web UI: go to your branch → "Compare & pull request".

---

## After Your PR is Merged

```bash
# Switch back to main and sync
git checkout main
git pull origin main

# Delete the merged branch locally
git branch -d feature/my-feature

# Delete the remote branch (optional — GitHub auto-deletes on merge if configured)
git push origin --delete feature/my-feature
```

## What NOT to Push

These should already be in `.gitignore`, but double-check before committing:

| Item | Reason |
|------|--------|
| `.env` | Contains secrets (Grafana password, API keys) |
| `mlruns/` | 1+ GB MLflow tracking data — tracked separately if needed |
| `data/raw/` | Source data — not version-controlled here |
| `data/processed/` | Derived data — regenerated by training pipeline |
| `*.pkl` model files > 100MB | Use Git LFS or store externally |
| `__pycache__/`, `*.pyc` | Build artifacts |

```bash
# Verify .gitignore is working
git status --short | grep -E "mlruns|\.pkl|\.env"
```

If you accidentally staged a large or sensitive file:

```bash
git reset HEAD <file>       # unstage without losing changes
git rm --cached <file>      # remove from tracking entirely
```

## Checking What Will Be Committed

```bash
git status                  # overall state
git diff --staged           # staged changes (what goes into commit)
git diff                    # unstaged changes
git log --oneline -10       # recent commit history
```

---

## Useful Shortcuts

```bash
# See full diff of last commit
git show HEAD

# Undo last commit (keep changes staged)
git reset --soft HEAD~1

# Stash work in progress
git stash
git stash pop

# Update your branch with latest main (rebase keeps history clean)
git fetch origin
git rebase origin/main
```

## Commit Message Convention

```
<type>: <short description (max 72 chars)>

[optional body — WHY this change was needed]
```

Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `style`

Examples:
```
feat: add searchable occupation combobox to predict form
fix: filter MLflow fallback warnings from home page warning bar
docs: add runbook for Docker operations
chore: split dev dependencies into requirements-dev.txt
```
